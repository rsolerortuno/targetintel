# v0.3.0 grounded extraction staging

This specification defines the first interpretation boundary for TargetIntel-IO
v0.3.0. It converts one explicitly supplied source document and one successful
structured LLM response into auditable, immutable *staging candidates*. It is
not evidence validation, target validation, citation verification, therapeutic
advice, score calculation, ranking, or EvidenceItem finalization.

## Contracts and schema

The implementation reuses the Issue 301 provider-neutral `LLMRequest` and
`LLMResponse` contracts, including canonical JSON, deterministic payload
identities, provider/model provenance, response statuses, and credential-key
rejection. The Issue 302 Ollama adapter can receive the schema through
`GroundedSchemaRegistry`; this layer itself never calls a provider. It does not
write Issue 303 execution artifacts or change their semantics.

The only schema is `targetintel.grounded_extraction` version `1.0.0`. Its
closed top-level object requires `schema_id`, `schema_version`,
`source_document_id`, and ordered `claims`. A claim is also closed and requires
`claim_text`, `quoted_span`, `quote_start`, `quote_end`, and a stance of
`supports`, `contradicts`, `contextual`, or `unclear`. Raw observations may be
target/disease/species/model-system mention lists and cohort, effect,
limitation, locator, or evidence-type strings. These remain raw observations;
they are not evidence vocabularies.

Unknown schema or schema version fails closed. Schema serialization is canonical
and resolver results are defensive copies. Hidden-reasoning keys (`thinking`,
`reasoning`, `chain_of_thought`, `scratchpad`, and `analysis`) are forbidden.

## Prompt and source boundary

The deterministic prompt is `targetintel.grounded_extraction.prompt` version
`1.0.0`. Its instructions explicitly treat the separately delimited
`<source_document>` as untrusted data, direct the model to ignore embedded
instructions, prohibit external knowledge and invented identifiers/citations,
and require schema-only output with no hidden reasoning. It permits an empty
claim list. The builder preserves `source_text` exactly and never truncates,
retrieves, or adds provider configuration. Prompt-injection protection here is
limited to instruction separation and this explicit source boundary.

The source-content hash is SHA-256 of exact `source_text` UTF-8 bytes. For the
existing structured-source request form it is SHA-256 of existing canonical JSON
serialization. No second canonicalization implementation is introduced.

## Grounding, identities, and audit outcome

Every accepted candidate is a distinct `GroundedClaimCandidate`, never an
`EvidenceItem`. Its quote is non-empty and must match `source[start:end]`
exactly: offsets are integer, non-negative, ordered, and in range; matching is
case-, whitespace-, punctuation-, and Unicode-sensitive. There is no fuzzy
matching, quote repair, external lookup, or omitted-offset recovery.

Candidate identity is locally computed from extraction format version, source ID
and content hash, exact offsets and quote, claim text, stance, and meaningful raw
fields. It excludes provider/model, request/response identities, timestamps,
paths, timing, token counts, and run IDs. Those excluded values remain audit
provenance on the candidate, not scientific identity. Nested raw fields are
deeply immutable and serialization is canonical JSON.

`GroundedExtractionResult` separates accepted candidates from sanitized rejected
records. Its deterministic identity covers format version, request and response
deterministic identities, source hash, ordered candidate IDs, ordered rejection
codes/safe fields, and extraction status; it excludes operational timing and
paths. Rejection messages contain no full source, quote, claim text, arbitrary
model values, traceback, or hidden reasoning.

The parser fails closed for non-success responses, missing structured output,
invalid top-level shape, schema/source mismatch, and non-array claims. Claim
errors remain visible with stable codes including `unknown_claim_field`,
`hidden_reasoning_field`, offset and quote failures, `invalid_stance`,
`malformed_optional_field`, and `duplicate_candidate`. Within one response,
the first exact candidate identity is accepted and later matches are rejected;
there is no cross-document or EvidenceItem duplicate processing.

An empty valid claims list is a successful `no_claims` extraction with no
rejections. It is scientifically neutral: it is not negative evidence or a
null biological result and cannot alter a target score. Citation verification,
evidence-family assignment, duplicate resolution beyond this response, human
review, and EvidenceItem finalization are explicitly deferred.
