# v0.4.0 modality-feasibility integration

`targetintel.modality` remains the only canonical public surface for modality
evaluation. Its legacy public API is `assign_modality_fit(...) -> ModalityCall`
and `annotate_modality_dataframe(...) -> pandas.DataFrame`. `ModalityCall`
contains legacy fit labels, a poor-direct-target flag, `best_modality`, and a
rationale. It deterministically evaluates a symbol plus optional role,
direction, and resistance-axis inputs. It has no target identity or Open
Targets tractability input.

Legacy API signatures, fields, labels, values, identities where present, and
ordering are unchanged. The legacy `best_modality` text is never
reinterpreted. This issue adds no second evaluator, ranking, feasibility score,
selector, profile builder, report, card, CLI output, or recommendation engine.

`compose_modality_with_feasibility` is additive. It takes an existing
`ModalityCall`, a validated `TargetFeasibilityProfile` or successful
`TargetFeasibilityProfileBuildResult`, an explicit modality, and explicit
target identifier/type because the legacy call has no identity. It returns an
immutable `ModalityAssessmentWithFeasibility` containing the untouched legacy
call and an immutable `ModalityFeasibilityAnnotation`.

Only the Issue 401 vocabulary (`antibody`, `small_molecule`, `protac`, and
`other_clinical`) is accepted. Explicit legacy aliases are `small molecule`,
`small-molecule`, and `other clinical`; ambiguous values such as
`IO-combination` fail closed. Invalid inputs, failed build results, unsupported
profile versions, bad coverage, and identity mismatches return sanitized,
deterministic composition errors.

The annotation stores source-linked observation IDs and Issue 403 coverage
contracts, not full payloads. It separates observations explicitly tagged with
the requested modality from `modality=None` target-level context for each of
tractability, clinical precedence, doability, and safety. Target-level context
is never relabelled as modality-specific. Clinical context is not universal
validation or a modality recommendation. Tractability is selected only for the
requested modality: small-molecule evidence never implies PROTAC feasibility.
Doability is never inferred from tractability or precedence. Empty, missing,
or failed safety retrieval is never represented as safety. Contradictions stay
visible.

Coverage reuses Issue 403 states and identities. For a direct profile without
build-result coverage, conservative terminal Issue-403 coverage records make
the absence visible; these are not confidence, a percentage, negative
evidence, or a ranking weight. Every supported dimension has requested-modality
and target-context coverage visibility.

Annotation IDs include version, target/type, modality, profile/source release,
canonical observation and coverage IDs, contradictions, limitations, and the
research-only/no-score/no-ranking/no-recommendation boundaries. Composite IDs
also include a canonical legacy assessment digest. Timestamps, paths, caches,
process data, and HTTP metadata are excluded. References are ordered by
dimension, modality, factor, source record, and observation ID.

Composition is pure: it performs no scoring, ranking, role classification,
feature or benchmark mutation, LLM work, transport/cache access, profile
construction, evidence persistence, synthesis construction, network access,
file output, subprocess, dynamic import, or unsafe deserialization. It changes
no original score, rank, label, or order and generates no clinical, safety,
target, modality, or go/no-go recommendation. Issue 405, not this issue, may
render this post-ranking decoration.
