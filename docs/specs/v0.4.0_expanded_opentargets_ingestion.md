# v0.4.0 expanded Open Targets ingestion

Issue 402 adds a source-level, injectable Open Targets ingestion boundary to
the Issue 401 feasibility contracts. It creates neither `TargetFeasibilityProfile`
objects nor feasibility scores, rankings, roles, modality advice, or LLM output.
The existing `targetintel.opentargets` association helper remains unchanged.

Two explicit request modes exist. `association_ranked` requires a disease ID and
uses bounded, ordered GraphQL pages. It retains source association score and
order as observations only. Reaching the requested page limit without an empty
page or reported total completion is `truncated`, never complete coverage.
`directed_target_universe` requires a non-empty explicit universe and resolves
and fetches every supplied target independently of association rank or score.

Gene symbols are upper-cased using canonicalization v1, then matched only to an
exact returned approved symbol. No aliases, synonym expansion, fuzzy matching,
or first ambiguous hit are accepted. Ensembl IDs use the `ENSG` plus eleven
digits structural form. The universe hash covers identifier type, sorted unique
canonical targets, and canonicalization version, so caller order cannot affect
identity and symbols cannot share an Ensembl cache identity.

Query plans are deterministic, contain only versioned project GraphQL documents,
and make no network or cache calls. The public live transport uses only the
configured official HTTPS GraphQL endpoint; it sends no caller headers, follows
no redirects, and tests use `FakeOpenTargetsTransport` rather than a server.
Raw transport payloads have a canonical SHA-256 identity and preserve GraphQL
errors and partial payloads explicitly. Release verification is `verified`,
`declared_unverified`, `not_reported`, or `mismatch`; a required mismatch fails
closed. A source that does not report a release is not claimed verified.

Directed coverage gives every requested target one terminal category: retrieved,
no record, unresolved, ambiguous, invalid, or retrieval failed. Its integer
numerator and denominator are explicit and coverage is not scientific confidence.
Empty safety arrays and missing safety fields are retained as source data; no
absence is converted to a safety conclusion.

Cache identity covers cache schema, release and verification state, endpoint,
mode, query version/document hash, disease/scope, pagination, identifier type,
and universe hash. `OpenTargetsCache` has an explicit root, validates canonical
payload hashes on read, and atomically replaces only its selected entry. Cache
location and timestamps are operational metadata and never record identity.

The GraphQL boundary is deliberately single-entity/sequential. Large systematic
universes should use a future release-pinned Open Targets bulk backend; that
backend can retain these request, coverage, and future profile contracts without
changing scoring or ranking systems.

## Compatibility and operational details

The existing `targetintel.opentargets` association helper remains the
compatibility boundary. Its low-level HTTP POST operation is narrowly shared
with this transport so the project has one live Open Targets HTTP
implementation. The legacy helper retains its public behaviour; the v0.4
transport fixes the official endpoint and disables redirects.

Search is candidate discovery, not symbol validation. For symbol requests each
candidate target document is retrieved and the source `target.approvedSymbol`
field must exactly equal the canonical requested symbol. The no-I/O planning
stage records source-independent operations only; the immutable execution plan
in the fetch result records every candidate-specific target-document operation
actually sent, and its expected operation count exactly equals that operation
list. No returned search `name` is treated as an approved symbol.

The current versioned association document does not expose a direct/indirect
association parameter. Accordingly, `association_scope` is rejected until a
future project-owned query version can actually send and retain that scope; it
is never accepted merely as cache metadata.

Cache entries retain request and plan identities, query type, target-universe
hash, requested release, verification state, response payload hashes, result
identity, and an operational creation timestamp. Reads revalidate the entry and
each response payload hash. Identity changes create new entries; corrupt or
mismatched entries are not reused. `cache_only` returns a closed error, whereas
`read_through` can refresh from live transport without deleting an unrelated
entry.

Missing safety fields remain missing source fields and an empty
`safetyLiabilities` array remains empty source data. No-record, malformed
response, and transport failure stay distinct terminal states and generate no
safety conclusion.
