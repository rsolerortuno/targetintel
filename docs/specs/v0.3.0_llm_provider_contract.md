# v0.3.0 provider-neutral LLM contract

## Boundary

This contract is the narrow boundary between explicitly supplied source content
and a future grounded-extraction layer. `LLMProvider.generate(LLMRequest)`
returns one `LLMResponse`. It neither retrieves documents nor writes storage,
finalizes `EvidenceItem` records, verifies citations, assigns evidence families,
or affects deterministic scores, rankings, benchmarks, or reports.

Provider output is therefore not an `EvidenceItem`, and provider success is not
scientific validation. A later layer must validate, ground, and explicitly
finalize any candidate evidence independently.

## Request and response lifecycle

Callers construct an immutable `LLMRequest` with a stable request ID, task and
source-document IDs, explicitly supplied text or structured source content,
versioned prompt and response-schema identifiers, instructions, and neutral
model configuration. Empty identifiers/instructions and absent source content
are rejected. Mappings are recursively frozen and mutable inputs are copied.
Credential-shaped configuration keys are rejected; API keys do not belong in
this contract.

The provider returns an immutable `LLMResponse`, whose `ProviderProvenance`
copies the request, prompt, schema, task, and source identifiers together with
the provider/model identity and result status. Parsed output is retained beside
raw text, so malformed output remains auditable rather than becoming an empty
successful response.

`ProviderCapabilities` only describes neutral functionality: structured and
JSON Schema output, deterministic seeds, system instructions, token and model
version reporting, execution locality, and context size. Each value can be
unknown (`None`); no vendor request parameters are encoded.

## Identity and serialization

All public contracts carry `llm-provider-contract-v1`, serialize through
canonical JSON with sorted keys and finite JSON numbers, and expose a SHA-256
`payload_identity`. Request identity covers the supplied deterministic payload.
Response identity excludes operational fields (`responded_at` and `latency_ms`)
while retaining deterministic output and provenance. Timestamps, where used,
must be timezone-aware. `to_dict`/`from_dict` support round trips.

Operational data is deliberately separate from payload identity: a clock may
record when a simulator call happened, but changing that time cannot change the
response payload identity.

## Status and errors

Responses use exactly one of `success`, `malformed_output`,
`retryable_failure`, `permanent_failure`, `timeout`, or `not_executed`.
Unsuccessful responses require an audit-facing error category. Rate limits,
timeouts, connection failures, and retryable provider failures are retryable;
the `retryable` property makes that policy explicit. Error messages are short
and sanitize common authorization/API-key/token/secret/password forms. Raw
exception representations, tracebacks, HTTP bodies, and credentials are not
contract data.

## MockProvider

`MockProvider` is an offline fixture simulator. It maps request IDs to frozen
`MockOutcome` values, never uses a network, filesystem, subprocess,
environment variable, model, or SDK, and never generates scientific content.
The script can model success, malformed output, retryable or permanent failure,
timeout, and non-execution. An absent request ID deterministically returns
`not_executed`. Its immutable tuple call-history view keeps the exact request
and response. An optional injected timezone-aware clock supplies operational
timestamps; fixture token and latency values are copied verbatim.

Future Ollama, Anthropic, OpenAI-compatible, and Gemini adapters can implement
this same synchronous protocol while keeping their SDK configuration outside
these contracts. This issue deliberately includes none of those SDKs or API-key
handling. The stable provenance and identity split support later grounded
extraction, reproducibility, and audit without treating model output as
scientific truth.
