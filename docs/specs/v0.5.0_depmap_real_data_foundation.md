# v0.5.0 DepMap real-data foundation

## Scope and scientific objective

v0.5.0 will eventually ask whether a target is functionally required by tumour
models and whether that dependency is enriched in melanoma rather than generic
or pan-essential. It must distinguish melanoma and non-melanoma dependency,
pan-cancer context, lineage selectivity, common/pan-essential controls,
heterogeneity, missing data, and insufficient model coverage. Issue 501 makes
none of those calculations or claims. It is research-only and does not validate
targets, biomarkers, mechanisms, or treatment choices.

The [current-state audit](../audits/v0.5.0_depmap_current_state.md) establishes
that no existing DepMap implementation exists. The canonical future surface is
therefore `targetintel.functional_dependency`; its contracts are domain models,
not operational ingestion. It has no client, downloader, complete-matrix parser,
dependency profile, data command, or score.

## Release-pinned manifests

Every execution supplies an explicit `DepMapReleaseManifest`, never `latest`.
The currently planned `DepMap Public 26Q1` is an example gate candidate, not a
hard-coded value. Source name, explicit release identifier, declaration state,
ordered file manifests, required/optional roles, schema fingerprints,
limitations, and research-use boundary form deterministic identity. Checksums
and schemas participate; local roots, timestamps, and operational metadata do
not. A requested release or layout-release mismatch fails closed. There is no
portal lookup, filename/directory inference, or fallback release.

`DepMapFileManifest` retains format version, controlled role, safe relative
filename, format, required state, SHA-256, expected bytes, schema fingerprint,
source description, and limitations. Absolute, traversal, URL, unknown-format,
malformed-hash, and negative-size inputs are rejected. A release rejects
duplicate roles, file identities, and filenames, and requires every declared
required role. Explicitly absent optional roles are exposed as
`unavailable_optional_roles`, never fabricated.

The controlled roles are `crispr_gene_effect`, `crispr_dependency_probability`,
`model_metadata`, `common_essential_reference`, `pan_dependency_reference`, and
`release_readme`. Roles express purpose rather than portal filenames. Initial
required core: gene effect, model metadata, and release README; probability and
reference controls may remain optional when unavailable.

`DepMapSchemaFingerprint` identifies role, orientation, identifier fields,
canonical required columns, gene/model label contracts, primitive value type,
nullable policy, and mapping version. Its ID changes with mapping changes.
CRISPR effect and probability are separate models-by-genes float schemas;
metadata is row-oriented `ModelID`; references are gene-oriented; README is a
release document. No complete matrix column list is stored; a future wide
matrix may use a deterministic header digest. Arbitrary caller schema shapes
are rejected against role-specific contracts.

## Local layout and validation

`DepMapLocalLayoutRequest` requires explicit absolute external and derived
roots, safe explicit release directory, cache policy, requesting actor, and
optional UTC timestamp. The conceptual paths are
`data/external/depmap/<release>/` and `data/derived/depmap/<release>/`; no
directories are created here. Paths and timestamps do not form scientific
identity. There is no home, cwd, or environment-variable discovery.

`validate_local_release` is offline and fixture-scale: it checks declared local
file presence, bytes, SHA-256, and one text header. Its immutable result
distinguishes valid, invalid-manifest, missing-role/file, checksum/size/schema/
release mismatch, unsupported format, invalid path, and validation failure with
sanitized errors. Validity does not imply gene coverage, data quality, melanoma
coverage, or biology. Production matrices are not parsed.

## Identity and future boundaries

Future gene handling retains original matrix label, parsed approved symbol when
available, and encoded Entrez ID separately. Rules are versioned; there is no
fuzzy match, alias acceptance, silent repair, or ambiguity resolution. Future
model mapping retains explicit `ModelID` (not row position), rejects duplicates,
exposes metadata-absent models, and selects melanoma via versioned explicit
lineage, disease, subtype, and OncoTree-like fields—not free-text matching.

Distinct future states are: file unavailable; file invalid; target absent;
target unresolved; metadata-absent model; no melanoma models; insufficient
melanoma models; measured/no dependency; missing gene effect; missing
probability; unavailable common-essential/pan-dependency reference; and
computation failure. None collapses to “no dependency”.

Issue 502 specifies explicit `full_matrix` and `target_subset` modes. Full
matrix is needed for genome-wide distributions, percentiles, pan-cancer
comparison, and lineage selectivity. Subset requires an explicit target
universe; its order-independent hash forms identity and cannot masquerade as
full-matrix output. Neither is implemented here.

Issue 504 will define three universes: an approximately 56-target curated
benchmark (validation only), a 300–1,000-gene discovery universe whose
membership does not use DepMap outcomes, and a genome-wide statistical
background. Benchmark labels never become model features.

## Isolation, security, and release gate

This package imports none of scoring, ranking, role classification, feature
construction, modality, Open Targets, LLM, cards, or reports. It cannot alter
or invoke them. It performs no network, subprocess, dynamic import, shell,
unsafe deserialization, or `eval`; recursive credentials and hidden reasoning
are rejected. No dependency is added.

Issue 502: local ingestion/header schemas. Issue 503: gene/model identity and
missingness. Issue 504: universes/cache identities. Issue 505: descriptive
dependency computation. Issue 506: isolated presentation. Issue 507: release
gate/reproducibility review.

Fixtures are mandatory for CI but cannot close v0.5.0. Issue 507 must pin a
real release; validate checksums and schemas; include real metadata, gene-effect,
and available probability data; demonstrate benchmark/discovery coverage and
genome-wide background statistics; create reproducible derived outputs; and
state explicit limitations. Until then, fixtures support no real-data closure.
